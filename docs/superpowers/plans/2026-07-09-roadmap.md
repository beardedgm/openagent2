# Brokerage Workspace — Delivery Roadmap

**Spec:** `docs/superpowers/specs/2026-07-09-brokerage-workspace-design.md`
**Feature detail:** `workspace_prd_v2.md` Phase 1 (sections 5.1–5.10)

Five stages. Each stage has (or will get) its own detailed implementation plan in this folder, written when the prior stage completes so it can reference real code. Every stage ends with working, tested software.

## Stage 1 — Foundation ✦ plan: `2026-07-09-stage-1-foundation.md`

Monorepo tooling (npm workspaces, ESLint, Prettier, Vitest, CI) · Express app skeleton (helmet, pino, sessions, error handling) · scrypt password util · Mongo models: Settings (singleton), User, Invitation, EngagementEvent · sliding-window rate limiter · Zod validate middleware · Turnstile helper (env-gated) · auth routes (login/logout/me/register-via-invite) · requireAuth/requireRole · email service (Resend/console) · invitation service + user management routes · storage adapter (local + R2) + logo/avatar uploads · settings routes · seed script · client scaffold (Vite, Router, TanStack Query, Axios) · design tokens + primitives + DESIGN.md · login/register pages + guards · app shell (sidebar/header/branding/accent color) · dashboard placeholder, profile, directory · admin Users + Settings pages · production static serving · CI workflow + README.

## Stage 2 — Communications ✦ plan: `2026-07-10-stage-2-communications.md`

- **Message Board (PRD 5.3):** Posts model + CRUD routes (officeAdmin+ create), TipTap rich text (bold/italic/lists/links/images) with server-side sanitization (`sanitize-html`), image upload to storage, office targeting, pinning (max 3), `publishAt` scheduling, keyword search, per-post comment toggle, flat comments with author/admin delete.
- **Notification system (PRD 5.9):** Notifications model, notification service (in-app always + email per user prefs; overdue-task emails non-disableable), bell drawer UI with unread count, prefs UI on profile settings, read/read-all endpoints.
- **Activity Feed (PRD 5.2):** ActivityEvents emission from existing actions (announcement posted, agent joined, etc.), `GET /feed` aggregation (internal + RSS) with 20-item pagination + infinite scroll, All/Internal/External filter, admin pin-to-top (≤7 days).
- **RSS ingestion:** Agenda setup (first job infrastructure), hourly poll of ≤10 feeds (`rss-parser`), RssItems cache collection, render from DB only.
- **Deferred trigger from Stage 1:** "invitation accepted" notifies the inviting admin.

## Stage 3 — Coordination ✦ plan: `2026-07-10-stage-3-coordination.md`

- **Calendar (PRD 5.4):** CalendarEvents model (recurrence rules, RSVP subdocs), month/week/day views (default month), office vs. personal events, RSVP yes/no/maybe + creator summary, reservable resources defined in Admin Panel with conflict check on save, 24h/1h email reminders (opt-in, Agenda), mandatory events (visual flag + feed notification), rich-text description.
- **Tasks (PRD 5.7):** Tasks model with per-assignee completion subdocs resolved at assignment time, priorities, due dates, attachments (≤5 × 25MB, storage adapter), related resource link, My Tasks panel + page, completion notes, recurring tasks (daily/weekly/monthly via Agenda), task templates CRUD (admin), notifications (assignment in-app; email for High priority / due <48h; 24h reminder; overdue badge + email).
- **Onboarding (PRD 5.8.2):** auto-assign onboarding template on registration, homepage progress bar until complete, per-agent onboarding status in admin Users page, onboarding template selector in admin.

## Stage 4 — Content

- **Resource Hub (PRD 5.6):** Categories (2 levels) admin CRUD, Resources model (file ≤50MB or external URL, office targeting, version history array), signed-URL downloads (15-min expiry) with download engagement logging, keyword search + category/file-type filters, featured (≤6) tile row, bookmarks + My Resources view, "new resource in bookmarked category" notification.
- **Banner Ads (PRD 5.5):** Banners model (image ≤5MB or rich text, CTA, office targeting, start/end dates), admin CRUD + duplicate/reschedule, homepage slot with ≤3 active + 5s rotation, graceful collapse when empty, click tracking endpoint + engagement log, click counts in admin view.

## Stage 5 — Polish & Launch

- **Dashboard assembly (PRD 5.1):** real widgets (feed preview 5, events 5, pinned announcements 3, quick links, banner slot, welcome message, my-tasks summary) wired to Stage 2–4 data; admin homepage layout config (enable/disable/reorder) + welcome message (rich text) + quick links editors.
- **Engagement:** `pageView` logging; verify all five event types flow.
- **Retention jobs (Agenda):** activity events 90d, RSS items 30d, task history 2y.
- **Brokerage-wide notification defaults** admin section (PRD 5.10.2).
- **Accessibility pass:** keyboard nav, WCAG AA contrast, aria-labels audit.
- **Deploy:** Render service (root build/start), R2 bucket + public base URL, Resend domain, Turnstile keys, UptimeRobot pinger, Sentry/PostHog if keys provided; deployment runbook in `docs/`.

## Cross-Stage Wiring Notes (do not lose)

- Stage 1 logs `login`; Stage 4 logs `download` + `bannerClick`; Stage 3 logs `taskComplete`; Stage 5 logs `pageView`.
- `Settings.onboardingTaskTemplateId` (Stage 1 field) is consumed in Stage 3.
- `Settings.homepageLayout`, `welcomeMessage`, `quickLinks` (Stage 1 fields) get admin editors in Stage 5.
- Notification triggers arriving before the notification system exists (Stage 1's "invitation accepted") are implemented in Stage 2.
- Assignee resolution snapshot rule (PRD 5.7.2): members added after task creation are NOT auto-included.
- Deactivated users: locked out at `requireAuth` on next request; content retained (PRD 5.8.4).
